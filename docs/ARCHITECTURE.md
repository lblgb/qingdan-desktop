## 技术栈

- 桌面框架：Tauri
- 前端框架：React
- 语言：TypeScript
- 构建工具：Vite
- 状态管理：Zustand
- 数据校验：Zod
- 时间处理：dayjs
- 本地数据库：SQLite

## 设计目标

- 第一版先保证本地可用。
- 从一开始就保持模块边界清晰。
- 桌面能力调用通过 Tauri 命令边界统一收敛。
- 前端业务模块与宿主层实现解耦，便于后续增加提醒、托盘、同步等能力。

## 目录结构建议

```text
docs/
src/
  app/
  components/
  features/
    tasks/
  lib/
  stores/
  types/
src-tauri/
  src/
    commands/
    db/
    models/
```

## 前端目录说明

### `src/app`

- 应用入口、全局布局、路由和页面装配。

### `src/components`

- 通用组件，例如按钮、输入框、弹窗、空状态和列表容器。

### `src/features/tasks`

- 待办任务领域模块。
- 包含任务列表、任务表单、任务筛选和任务类型定义的前端部分。

### `src/lib`

- 基础工具函数、时间格式化、Tauri API 封装和通用常量。

### `src/stores`

- Zustand 状态管理。
- 负责界面状态与任务数据状态的编排。

### `src/types`

- 前端共享类型定义。

## Tauri 侧目录说明

### `src-tauri/src/commands`

- 对前端暴露的命令入口。
- 例如任务查询、新增任务、更新任务状态等。

### `src-tauri/src/db`

- SQLite 连接、初始化和数据访问封装。

### `src-tauri/src/models`

- Rust 侧数据结构和序列化模型定义。

- 编辑任务
- 删除任务
- 标记完成
- 任务列表展示
- 基础状态筛选
- 本地持久化

- 云同步
- 用户系统
- 多人协作
- 复杂提醒

- 统计分析

### 真实数据层接入范围

### 第一阶段目标

- 将当前前端 `localStorage` 占位实现切换为 `Tauri 命令层 + SQLite` 的真实本地持久化。
- 保持前端现有交互能力不变，只替换数据来源和写入链路。
- 继续维持“前端不直接耦合数据库细节、由命令边界统一收敛”的原则。

### 第一阶段仅包含

- 数据库初始化与任务表建表。
- 任务列表读取。
- 新建任务。
- 标记完成与取消完成。
- 删除任务。
- 前端状态仓库改为调用异步数据访问层。

### 第一阶段明确不包含

- 任务编辑。
- 任务排序规则重构。
- 任务分组与折叠结构。
- 数据迁移工具与 `localStorage` 自动导入。
- 标签、优先级、提醒、统计等扩展字段的正式落地。

### 第一阶段验收基线

- 桌面端启动后可以从 SQLite 读取任务而不是使用 `localStorage` 作为主数据源。
- 新增、完成、删除操作在重启应用后仍能保留。
- 前端任务模块不直接书写 SQL，也不直接依赖 SQLite 插件接口。
- 文档、目录边界和注释与实现保持一致。

### 任务排序与分组语义

### 设计目标

- 先定义稳定、可解释的任务展示口径，再进入第二阶段查询能力实现。
- 规则应服务于“每天打开就能快速决定先做什么”，而不是追求复杂视图。
- 排序与分组不新增数据字段，优先基于现有 `completed`、`dueAt`、`updatedAt` 语义工作。

### 第一版默认展示口径

- `进行中` 任务优先于 `已完成` 任务。
- 在进行中任务里，已设置截止日期的任务优先于未设置截止日期的任务。
- 在已设置截止日期的任务里，按截止日期从早到晚排序。
- 在截止日期相同或都未设置截止日期时，按最近更新时间倒序排序。
- 已完成任务默认沉底显示，不进入进行中任务的主决策区。

### 第一版分组口径

- 仅对 `全部任务` 和 `进行中` 视图提供分组语义。
- 第一版分组采用时间语义分层，而不是标签或优先级分层。
- 分组顺序固定为：
  1. `已逾期`
  2. `今天`
  3. `未来`
  4. `未安排`
- `已完成` 视图第一版不做多组分层，保持单组列表即可。

### 分组判定规则

- `已逾期`：任务未完成，且 `dueAt` 早于当前本地日期。
- `今天`：任务未完成，且 `dueAt` 等于当前本地日期。
- `未来`：任务未完成，且 `dueAt` 晚于当前本地日期。
- `未安排`：任务未完成，且 `dueAt` 为空。
- `已完成`：只由 `completed = true` 判定，不再参与上述时间分组。

### 责任边界

- 第二阶段先由命令层返回“已按默认口径排序的扁平任务列表”。
- 分组结构先在前端基于统一规则计算，避免命令层过早承担视图组织职责。
- 当后续出现多视图、多排序方式或统计需求时，再评估是否将分组结果下沉到命令层。

### 第一版明确不包含

- 用户自定义排序。
- 拖拽排序。
- 标签分组、优先级分组、项目分组。
- 已完成任务自动折叠交互。
- 多套视图并行存在时的独立排序偏好持久化。

### 第二阶段验收基线

- 同一批任务在重启应用后仍按同一展示口径排序。
- 前端所有视图共享同一套分组判定规则，不出现口径漂移。
- 命令层与前端对“逾期 / 今天 / 未来 / 未安排 / 已完成”的语义保持一致。
- 文档先更新，再进入排序与分组的代码实现。

### 任务编辑接入范围

### 本轮目标

- 补齐第一版功能范围内尚未落地的任务编辑能力。
- 保持当前列表视图为主，不新增独立详情页或复杂弹窗流。
- 编辑完成后继续复用现有排序、分组与 SQLite 持久化链路。

### 本轮仅包含

- 在任务列表中对单条任务进行内联编辑。
- 支持修改标题、备注和截止日期。
- 编辑提交后立即写入 SQLite，并刷新当前列表。
- 保持当前筛选、排序和分组口径不变。

### 本轮明确不包含

- 批量编辑。
- 独立任务详情页。
- 富文本备注。
- 编辑历史或撤销/重做。
- 编辑时调整完成状态以外的新字段。

### 交互基线

- 每次仅允许一条任务进入编辑态。
- 编辑态应保留“保存”和“取消”两个明确动作。
- 若标题为空，不允许提交。
- 取消编辑时不应修改原始任务内容。

### 验收基线

- 编辑后的标题、备注和截止日期在应用重启后仍能保留。
- 编辑完成后，任务应继续遵循当前排序与分组语义重新落位。
- 编辑逻辑不破坏现有完成、删除和筛选能力。

### 第一版收口范围

### 本轮目标

- 在既有任务流不扩边界的前提下，补齐用户可见的加载中、保存中、成功和失败反馈。
- 对第一版当前代码执行正式构建验证，确认至少能完成前端生产构建和宿主层 release 编译。
- 把正式打包链路中的外部依赖风险显式记录到文档，而不是隐含在命令执行结果里。

### 本轮仅包含

- 状态仓库补充异步动作态与反馈态。
- 在首页、录入区和任务列表中展示加载与操作反馈。
- 失败场景下提供最小可用的重试入口或关闭提示入口。
- 执行 `tsc -b`、`cargo check` 和 `tauri build` 级别的构建验证。
- 补齐 Tauri 发布配置中的产品名、窗口标题和图标引用，避免发布产物继续依赖默认或空配置。

### 本轮明确不包含

- 新的任务字段。
- 排序与分组规则调整。
- 正式品牌图标替换。
- 安装包分发流程、自动更新和版本升级策略。

### 验收基线

- 用户在界面内可以感知本地任务加载中、写入中、成功和失败状态。
- 任务新增、编辑、完成和删除在异步过程中有明确反馈，不再只依赖静默等待。
- `tauri build` 至少能走到宿主层 release 可执行文件生成阶段。
- 若正式打包受外部依赖阻塞，需在文档中明确阻塞点与当前替代验证结果。
- 第一版手工验收应统一以 `docs/V1_ACCEPTANCE.md` 为执行清单，避免每轮收口时口径漂移。

### `v0.1.5` 规划范围

#### 版本目标

- `v0.1.5` 不再以“补齐基础 CRUD”为主，而是进入“任务组织能力升级”。
- 本轮目标是在不引入复杂项目系统的前提下，补齐“弹窗新建任务”和“任务组”两条能力线。
- 设计上继续保持“先写清文档口径，再按文档实现”的推进方式，避免组模型与筛选语义边写边漂移。

#### 本轮纳入范围

- 将首页常驻的任务录入区改为“按钮触发 + 弹窗表单”。
- 新增 `任务组` 数据模型，并接入 SQLite 持久化。
- 支持组的基础 `CRUD`。
- 支持任务归组与移出组。
- 支持新建任务时直接选择所属组。
- 将现有筛选扩展为“状态筛选 + 组筛选”并存。

#### 任务组模型基线

- 一个任务在 `v0.1.5` 中最多只允许属于一个组。
- 一个组可以包含多个任务。
- 组的最小字段先定义为：
  - `id`
  - `name`
  - `description`
  - `createdAt`
  - `updatedAt`
- 本轮暂不引入组颜色、组图标、组排序权重和归档状态。

#### 删除组的处理规则

- 删除组时，不删除组内任务。
- 原组内任务统一转为“未分组”。
- 该规则优先于“级联删除”方案，因为它更适合当前个人待办工具的低风险预期。

#### 新建任务弹窗交互基线

- 首页主内容区保留“新建任务”明确入口，但不再常驻展开表单。
- 点击入口后，以弹窗承载标题、备注、截止日期和所属组选择。
- 弹窗需保留当前第一版已有的保存中、失败提示和禁用态能力。
- 支持 `Esc` 关闭弹窗。
- 支持通过表单提交动作完成保存，不要求在本轮强行加入复杂快捷键体系。

#### 筛选与展示语义

- 状态筛选继续保持三档：
  - `全部任务`
  - `进行中`
  - `已完成`
- 新增组筛选维度：
  - `全部组`
  - `未分组`
  - `某个具体组`
- 两层筛选应同时生效，最终结果为“状态筛选结果”与“组筛选结果”的交集。
- 第一版已存在的时间排序和时间分组口径继续保留，组筛选只是缩小任务集合，不替代时间语义。

#### 责任边界

- 命令层负责组与任务关系的真实持久化能力，以及按筛选条件返回任务列表。
- 前端状态仓库负责组合“状态筛选 + 组筛选”的界面状态。
- 前端列表仍可继续基于统一时间语义做展示分组，避免命令层过早返回带视图结构的复杂数据。

#### 本轮明确不包含

- 一个任务属于多个组。
- 组内嵌套子组。
- 组拖拽排序。
- 组折叠状态持久化。
- 标签、优先级、提醒、日历视图与任务组一起并入同一轮实现。
- 独立的“组详情页”或复杂项目面板。

#### 建议实现顺序

1. 先补组的数据模型、SQLite 表结构和基础命令层接口。
2. 再补前端状态仓库中的组列表、组筛选和任务归组语义。
3. 然后把首页录入区切换为按钮 + 弹窗。
4. 最后再收列表中的组筛选展示、空状态和交互反馈。

#### `v0.1.5` 验收基线

- 可以创建、编辑和删除任务组。
- 可以在新建任务时选择所属组。
- 可以把现有任务调整到某个组，或移回未分组。
- 删除组后，原组内任务仍保留，且变为未分组。
- 状态筛选与组筛选可组合工作，不出现结果口径冲突。
- 文档先更新，再进入 `v0.1.5` 的代码实现。
- 第二版手工验收应统一以 `docs/V015_ACCEPTANCE.md` 为执行清单，避免收口时再次回到口头验收。

## `v0.20` 规划范围

### 版本目标

- 在不推翻 `v0.1.5` 主界面骨架的前提下，补齐优先级、更多条件、轻量批量操作、任务概览增强和系统级界面精修。
- 补齐 `快捷操作强化`，让全局入口、列表动作和任务级轻量操作保持更短路径。
- 本轮目标不是改造为全新产品形态，而是把现有任务决策、整理和可见性能力做稳健扩展。
- 继续坚持“先写清文档基线，再按文档实现”的推进方式，避免在实现过程中隐式扩边界。
- `v0.20` 不是对 `v0.1.5` 的重做，而是围绕“优先级、更多条件、轻量批量、概览增强和系统级精修”的稳健扩展。

### 本轮纳入范围

#### 1. 优先级正式落地

- 为任务新增正式字段 `priority`。
- 固定四档优先级：
  - `urgent` -> `紧急`
  - `high` -> `高`
  - `medium` -> `中`
  - `low` -> `低`
- 新旧任务都必须具备明确优先级。
- 数据库升级后，历史任务默认补为 `medium`。
- `priority` 使用稳定枚举值，不使用自由文本。
- 命令层与前端展示需要共享同一套优先级语义，避免中文展示和存储枚举不一致。
- 兼容迁移必须先补列或补默认值，再执行依赖该字段的查询、排序和写入逻辑。

#### 2. 更多条件体系扩展

- 左侧 `更多条件` 扩展为四类：
  - `任务组`
  - `优先级`
  - `时间范围`
  - `排序方式`
- 主筛选仍然只负责：
  - `全部任务`
  - `进行中`
  - `已完成`
- `任务组` 和 `优先级` 继续作为附加决策维度，不替代任务状态。
- `时间范围` 与 `排序方式` 继续保持固定枚举口径，不做自由组合。
- 主筛选 + 更多条件的最终结果定义为交集：先由主筛选限定状态集合，再由更多条件继续缩小结果集。
- 过滤语义保持交集，不允许任意一层筛选单独覆盖另一层筛选的结果。
- 时间分组和更多条件筛选是两条不同语义线，前者用于列表展示分层，后者用于收窄任务集合。

#### 2.1 枚举口径

- `任务组`：
  - `全部组`
  - `未分组`
  - `某个具体组`
- `优先级`：
  - `全部优先级`
  - `紧急`
  - `高`
  - `中`
  - `低`
- `时间范围`：
  - `全部时间`
  - `今天`
  - `未来 7 天`
  - `已逾期`
  - `无日期`
- `排序方式`：
  - `默认推荐`
  - `截止日期优先`
  - `优先级优先`
  - `最近更新`

#### 2.2 排序语义

- 默认推荐延续当前时间语义：
  - 进行中优先于已完成
  - 有截止日期优先于无截止日期
  - 截止日期更早的优先
  - 截止日期相同或无截止日期时，按最近更新时间倒序
- 截止日期优先更强调时间驱动，适合尽早处理临期任务。
- 优先级优先先按 `紧急 / 高 / 中 / 低` 排序，再套时间语义规则。
- 最近更新以最近更新时间为主，适合回看刚处理过的任务。

#### 2.3 本轮限制

- 本轮只支持固定排序方式切换。
- 不支持用户自定义排序规则组合。
- 不支持独立的排序偏好管理面板。

#### 3. 轻量批量操作

- 提供列表多选模式。
- 默认列表不常驻复选框。
- 列表进入批量模式后，顶部工具条切换为批量栏。
- 进入批量模式后，允许：
  - 批量改优先级
  - 批量归组
  - 批量标记完成
- 本轮不纳入批量删除。
- 批量栏至少包含：
  - 已选数量
  - `改优先级`
  - `改任务组`
  - `标记完成`
  - `取消选择`
- 本轮明确不包含批量删除，也不包含批量编辑标题、备注和截止日期。

#### 4. 任务概览增强

- `任务概览` 继续以弹窗承载，不新增独立概览页面。
- 概览增强重点包含：
  - 优先级分布
  - 最近 7 天趋势
  - 本周周报卡片
- 趋势默认显示完成数，并保留切换到“新建 / 完成双线”的能力。
- 概览层继续保留周报和趋势作为内部模块，不上升为主界面常驻区域。
- 优先级分布用于展示 `紧急 / 高 / 中 / 低` 的数量与占比。
- 最近 7 天趋势默认展示完成数，并支持切换为“新建 / 完成双线”对比视图。
- 本周周报卡片至少展示：本周完成数、本周新增数、逾期变化、当前最高优先级未完成数。

#### 5. 系统级界面精修

- 优化字体层级。
- 优化按钮密度与节奏。
- 统一更多条件的结构与选中态表达。
- 强化列表状态表达。
- 将概览弹窗收敛为更稳定的信息面板风格。

#### 6. 快捷操作强化

- 全局层保留 `任务概览`、`任务组管理` 和 `新建` 三个高频入口。
- 列表层保留进入或退出批量模式、标记完成和进入编辑的轻量动作。
- 任务层保留轻量切换优先级的快捷路径。
- 本轮只强化高频动作路径，不引入复杂快捷键体系。
- 全局、列表、任务级动作保持轻量，不引入完整快捷键中心或多步命令流。

### 责任边界

- 命令层负责优先级补齐、更多条件查询口径、批量动作所需的最小数据接口，以及旧数据的兼容升级。
- 前端状态仓库负责筛选条件、排序方式、批量模式和概览开关的统一编排。
- 前端列表负责展示优先级、更多条件结果、批量选择态和任务概览入口，不把复杂查询语义散落到页面组件里。
- 概览依然是派生数据视图，和主列表共享同一套优先级与时间范围定义。
- 左侧信息架构继续以 `工作视图` 为主容器，`更多条件` 承载任务组、优先级、时间范围和排序方式。
- 右侧列表区域继续作为主工作区，列表顶栏承载当前条件摘要、当前排序方式和批量模式入口。
- `任务概览` 作为独立弹窗入口存在，不回收进主内容卡片。

### 本轮明确不包含

- 快速新建条。
- 标签系统。
- 多任务组归属。
- 子组或嵌套组。
- 拖拽排序。
- 批量删除。
- 日历主视图。
- 服务端同步。
- 多人协作。
- 用户自定义多字段排序组合。
- 独立的排序偏好管理面板。
- 复杂快捷键体系。

### `v0.20` 验收基线

- 任务可以设置并持久化 `urgent/high/medium/low` 四档优先级。
- 旧任务升级后默认优先级正确补齐。
- 左侧 `更多条件` 同时支持任务组、优先级、时间范围和排序方式。
- 状态筛选与更多条件可组合工作，结果口径一致。
- 列表可以进入批量模式。
- 可以完成批量改优先级、批量归组和批量标记完成。
- 批量模式退出后恢复普通浏览态。
- `任务概览` 可以展示优先级分布、最近 7 天趋势和本周周报卡片。
- 概览趋势默认显示完成数，并保留切换到新建/完成双线的能力。
- 全局、列表和任务级快捷动作路径保持简短且一致。
- 文档、工作日志与实现保持同步，不先实现再补文档。

### 实施顺序

1. 先更新架构文档与验收文档，定清 `v0.20` 语义基线。
2. 再补 SQLite 与命令层的数据模型、兼容迁移和查询参数。
3. 再扩前端状态仓库，统一筛选、排序和批量模式状态。
4. 再做列表中的优先级展示、更多条件、排序切换与批量操作。
5. 最后补任务概览增强与系统级界面精修。

### 风险控制

- 旧库必须安全补齐 `priority`，不能依赖删库。
- 前端、命令层、概览统计必须共享同一套优先级与时间范围定义。
- 严格保持“不做快速新建条、不做批量删除、不做复杂快捷键体系”的边界。
- 默认态保持克制，把复杂度放到进入筛选、概览或批量模式之后再暴露。

## 工程约束补充

- 前端不得直接耦合数据库实现细节。
- Tauri 命令层只暴露业务需要的最小接口。
- 数据模型需要预留扩展字段，例如优先级、截止时间和备注。
- 每个新增模块都应遵守中文注释和文件头说明要求。
