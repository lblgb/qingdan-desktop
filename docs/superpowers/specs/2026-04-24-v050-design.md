# `v0.50.0` 设计方案

## 1. 版本定位

`v0.50.0` 定位为一次同时覆盖产品观感、数据安全和规模化使用效率的版本升级。

本版不推翻轻单当前的主工作台结构，仍保留：

- 顶部动作区
- 左侧工作视图
- 中间任务列表
- 弹窗式概览、提醒、设置与任务详情

但会将整套界面语言升级为“深色科技控制台”风格，并补齐数据备份、导出与搜索定位能力。

## 2. 版本目标

- 将轻单从“常规桌面待办工具”升级为“深色科技控制台感”的任务工作台。
- 补齐本地优先产品必须具备的数据安全闭环：可备份、可恢复、可导出。
- 在任务数量增加后，提供可用的全局搜索与快速定位能力。

## 3. 版本范围

### 3.1 本轮纳入范围

1. 科技控制台 UI 改版
2. 备份与恢复中心
3. 数据导出
4. 全局搜索与快速定位

### 3.2 本轮明确不包含

- 云同步
- 多端同步
- 日历主视图
- 子任务系统
- 标签系统重构
- 自动定时备份策略
- 复杂搜索语法或命令面板系统
- CSV 导入
- PDF、图片等展示型导出

## 4. 任务拆解

### 任务 1：科技控制台 UI 改版

#### 目标

在不推翻当前信息架构的前提下，建立统一的深色科技控制台视觉语言。

#### 核心设计方向

- 主色基调为深海军蓝到近黑底色。
- 强调色以青蓝、电青为主，绿色仅用于成功或通过状态。
- 使用更硬朗的圆角、边框和局部发光，不走柔软玻璃拟物路线。
- 标题风格更像系统面板或数据终端，正文仍需保持高可读性。
- 动效只服务于状态感知，不做纯装饰性炫技。

#### 具体改造范围

- 顶部动作区改为“主控制条”。
- `任务概览 / 任务组管理 / 新建` 保留为高频主动作。
- `设置 / 提醒 / 备份` 收拢为右上角系统图标入口。
- `提醒` 图标右上角显示待处理数量角标。
- 左侧工作视图改造成更像“导航矩阵”的结构。
- 任务列表改造成更像“任务模块卡”的表达。
- 优先级、截止时间、任务组、提醒状态统一为状态芯片体系。
- 概览、提醒中心、设置、任务详情、任务组管理等弹窗统一面板语言。

#### 视觉关键词

- 深色控制台
- 边缘发光
- 细网格
- 系统状态条
- 冷静、精密、偏监控界面的信息表达

### 任务 2：备份与恢复中心

#### 目标

建立明确、可信、可操作的本地数据安全入口。

#### 入口策略

- 不放入设置子项。
- 在右上角提供独立图标入口。
- 使用轻量弹窗或面板承载，不先扩展成重型独立页面。

#### 最小闭环

- 手动创建备份
- 从备份文件恢复
- 展示最近备份时间与基础状态信息
- 恢复前风险确认
- 非法文件、读取失败、格式不兼容时统一错误反馈

#### 设计要求

- 备份中心要像“系统控制仓”，而不是单一表单。
- 强化“当前状态是否安全”的感知，而不只是罗列按钮。

### 任务 3：数据导出

#### 目标

提供“能带走”的可读、可迁移数据出口。

#### 入口策略

- 主出口放入备份与恢复中心。
- 快捷出口放入任务列表工具条，默认导出当前筛选结果。

#### 导出能力

- JSON 导出：
  - 面向完整保真
  - 适合迁移、存档、二次使用
- CSV 导出：
  - 面向表格查看与外部处理
  - 适合日常整理、汇报或人工加工

#### 导出范围

- 全量导出
- 当前筛选结果导出

#### 约束

- 不做 PDF
- 不做图片导出
- 不做模板化导出
- 不做 CSV 导入

### 任务 4：全局搜索与快速定位

#### 目标

在任务数量变多时，提供低阻力的检索和回到任务能力。

#### 入口策略

- 搜索入口放在顶部主控制区。
- 结果层采用轻量下拉面板。
- 不做完整命令面板系统。

#### 本轮能力

- 按标题搜索
- 按备注搜索
- 结果点击后自动切换必要筛选
- 自动滚动到目标任务并短暂高亮

#### 设计要求

- 搜索结果面板要保持“科技控制台”的视觉语言。
- 搜索结果应同时体现任务标题、状态、优先级和必要的时间信息。

## 5. 信息架构调整

### 顶部控制区

- 左侧保留版本主动作区。
- 右侧新增系统图标组：
  - 备份与恢复
  - 提醒
  - 设置

### 列表工具条

- 保留筛选、排序、批量模式相关控制。
- 增加“导出当前结果”的快捷出口。

### 数据安全入口关系

- 备份中心负责：
  - 备份
  - 恢复
  - 全量导出
- 列表工具条负责：
  - 当前筛选结果导出

## 7. 风险与控制

### 风险 1：UI 改动范围过大

控制方式：

- 严格限定在当前结构内升级视觉语言，不新增重型页面体系。
- 优先统一骨架和组件语言，再处理细节动效。

### 风险 2：恢复操作具备破坏性

控制方式：

- 恢复前强制确认。
- 恢复失败时不中断应用数据读取链路。
- 恢复成功后统一刷新任务状态。

### 风险 3：导出与备份边界混淆

控制方式：

- 在文案和入口层明确区分：
  - 备份是为了恢复
  - 导出是为了带走和外部使用

### 风险 4：搜索体验过重

控制方式：

- 首版只做轻量下拉搜索结果。
- 不引入快捷键体系和命令面板 DSL。

## 8. 验收口径建议

后续应新增 `docs/V050_ACCEPTANCE.md` 作为唯一手工验收清单，至少覆盖：

- 主工作台新视觉语言是否统一
- 顶部系统图标入口是否成立
- 提醒角标是否准确
- 备份创建与恢复闭环是否可用
- JSON / CSV 导出是否可用
- 搜索后自动定位和高亮是否稳定

## 9. 实施顺序建议

1. 先建立 `v0.50.0` 文档基线与验收清单。
2. 再做 UI 设计变量、图标入口和主控制区骨架调整。
3. 再落备份与恢复中心。
4. 再补导出能力。
5. 最后接入全局搜索与快速定位，并做整体视觉收口。
