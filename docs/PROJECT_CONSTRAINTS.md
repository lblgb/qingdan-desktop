# 项目开发约束

## 文件说明

本文档用于记录本项目当前已达成的开发约束，作为后续编码、评审、提交和功能推进的执行依据。

## 当前已确认约束

### 1. 对话与决策记录

- 项目内需要持续记录每一轮讨论中的关键信息。
- 记录内容至少包括：讨论主题、达成结论、主要原因、待确认事项。
- 建议持续维护 `docs/WORKLOG.md`，必要时再拆分出 `docs/DECISIONS.md`。

### 2. 中文优先

- 项目文档默认使用中文。
- 代码注释默认使用中文。
- 界面文案默认使用中文。

### 3. 注释规范

- 每个源码文件文件头需要包含简要注释，说明该文件的用途。
- 结构体、类型定义、核心函数需要添加简要注释。
- 对复杂逻辑可以按需补充更详细的说明性注释。
- 注释应以“帮助后续维护”为目标，避免无信息量的重复描述。

### 4. 功能完成后的版本同步

- 每个小功能完成后，需要形成独立提交。
- 提交完成后需要同步到 GitHub 远端仓库。
- 如果某个功能较大，应先拆分为多个可验证的小提交，再进行推送。

### 5. 第一阶段目标

- 先实现待办软件第一版可用版本。
- 第一版以本地可用为主，优先保证基础体验和后续扩展能力。
- 功能细节可以继续对齐，但工程结构要从一开始就为扩展预留空间。

### 6. 扩展性要求

- 目录结构需要清晰，避免页面逻辑、状态管理、数据访问混杂。
- 功能模块应尽量解耦，便于后续增加提醒、标签、同步、统计等能力。
- 数据模型设计需要考虑后续扩展字段，而不是只满足最小可跑。
- 提前定义基础约定，例如状态流转、数据存储层边界、桌面端能力调用边界。

### 7. 编码与配置文件约束

- 项目内文本配置文件统一使用 UTF-8 无 BOM，尤其是 `Cargo.toml`、`tauri.conf.json` 等会被构建工具直接解析的文件。
- 涉及 Tauri Windows 构建时，需保证 `src-tauri/icons` 下存在有效图标资源。
- 环境问题排查时应优先区分“系统依赖问题”和“项目配置问题”，避免混杂处理。

## 当前状态

### 1. Git 与仓库

- 当前工作目录已初始化为 Git 仓库。
- 仍需补充 GitHub 远端地址与首次推送流程。

### 2. 技术栈

- 已确认采用 `Tauri + React + TypeScript`。
- 前端当前已接入 `Zustand`、`Zod`、`dayjs`。
- 宿主层后续将继续围绕 Tauri 命令边界与 SQLite 持久化演进。

### 3. 当前阶段重点

- Tauri 开发环境已打通，桌面窗口已可启动。
- 下一阶段重点从“环境修复”切换为“UI/UX 打磨 + 真实数据层接入”。

### 4. 当前实现边界

- 真实数据层接入应按阶段推进，先实现与当前界面等价的基础 CRUD 持久化能力。
- 排序、分组、编辑、标签和优先级不在本轮 SQLite 接入范围内，需在数据边界稳定后单独设计。
- 功能实现前应先更新文档明确范围，再按文档基准推进代码。

- 远端仓库地址仍待补充。

## 变更原则

- 新约束在讨论确认后，应直接更新本文档。
- 如果新决策与旧约束冲突，应明确标注覆盖关系，而不是隐式替换。
