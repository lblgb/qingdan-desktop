# 项目开发约束

## 文件说明

本文档用于记录当前项目已经确认的开发约束，作为后续编码、评审、提交和功能推进的执行依据。

## 当前已确认约束

### 1. 对话与决策记录

- 项目内需要持续记录每一轮讨论中的关键信息。
- 记录内容至少包括：讨论主题、达成结论、主要原因、待确认事项。
- 默认持续维护 `docs/WORKLOG.md`，必要时再拆分出 `docs/DECISIONS.md`。

### 2. 中文优先

- 项目文档默认使用中文。
- 代码注释默认使用中文。
- 界面文案默认使用中文。

### 3. 注释规范

- 每个源码文件文件头需要包含简要注释，说明该文件的用途。
- 结构体、类型定义、核心函数需要保留简要注释。
- 对复杂逻辑可按需补充更详细的说明性注释。
- 注释应以“帮助后续维护”为目标，避免无信息量的重复描述。

### 4. 功能完成后的版本同步

- 每个小功能完成后，需要形成独立提交。
- 提交完成后需要同步到 GitHub 远端仓库。
- 如果某个功能较大，应先拆分为多个可验证的小提交，再进行推送。

### 5. 第一阶段目标

- 先实现待办软件第一版可用版本。
- 第一版以本地可用为主，优先保证基础体验和后续扩展能力。
- 功能细节可以继续对齐，但工程结构要从一开始就为扩展预留空间。

### 6. 扩展性要求

- 目录结构需要清晰，避免页面逻辑、状态管理、数据访问混杂。
- 功能模块应尽量解耦，便于后续增加提醒、标签、同步、统计等能力。
- 数据模型设计需要考虑后续扩展字段，而不是只满足最小可跑。
- 需提前定义基础约定，例如状态流转、数据存储层边界、桌面端能力调用边界。

### 7. 编码与配置文件约束

- 项目内文本配置文件统一使用 UTF-8 无 BOM，尤其是 `Cargo.toml`、`tauri.conf.json` 等会被构建工具直接解析的文件。
- 涉及 Tauri Windows 构建时，需保证 `src-tauri/icons` 下存在有效图标资源。
- 涉及 Tauri 发布配置时，需显式配置产品名、窗口标题和 `bundle.icon`，避免发布产物继续依赖空数组或默认值。
- 当 Windows MSI 需要承载中文产品名、安装目录和快捷方式名称时，需为 WiX 显式提供支持中文代码页的本地化配置。
- Windows 发布版应使用 `windows_subsystem = "windows"`，避免安装版启动时弹出额外控制台窗口。
- GitHub Release 发布时，默认应同时附带两类 Windows 安装产物：
  - `轻单_<version>_x64-setup.exe`
  - `轻单_<version>_x64_zh-CN.msi`
- 如某次发布缺失上述任一安装产物，应视为发布内容不完整，而不是仅凭 tag 或 release notes 视为已完成发布。
- 环境问题排查时应优先区分“系统依赖问题”和“项目配置问题”，避免混杂处理。
- Windows 下首次执行 `tauri build` 打 MSI 时，可能需要额外下载 WiX 工具链；若下载受网络影响，需把“release 可执行文件已生成”和“安装包打包被外部依赖阻塞”分开记录。

## 当前状态

### 1. Git 与仓库

- 当前工作目录已经初始化为 Git 仓库。
- GitHub 远端仓库已建立并完成首次同步。
- 后续继续按“功能完成后独立提交并推送”的节奏推进。

### 2. 技术栈

- 已确认采用 `Tauri + React + TypeScript`。
- 前端当前已接入 `Zustand`、`Zod`、`dayjs`。
- 宿主层继续围绕 Tauri 命令边界与 SQLite 持久化演进。
- 本机已补齐 `Python 3.12` 运行环境，后续依赖本地 Python 的 skill 和辅助脚本应优先复用现有环境。

### 3. 当前阶段重点

- Tauri 开发环境已打通，桌面窗口可正常启动。
- 当前主线已从“第一版功能收口”切换为 `v0.1.5` 的第二版规划与实现准备。
- `v0.1.5` 优先处理任务组织能力升级，不在开局阶段继续发散到提醒、日历和复杂项目系统。

### 4. 当前实现边界

- 真实数据层接入按阶段推进，先实现与当前界面等价的基础 CRUD 持久化能力。
- 排序、时间分组和编辑已按文档基线接入，`v0.1.5` 在此基础上继续扩展“任务组”和“弹窗新建”。
- 功能实现前应先更新文档明确范围，再按文档基准推进代码。
- `v0.1.5` 任务组语义当前已确认：一个任务最多属于一个组；删除组时保留组内任务并转为未分组。
- `v0.1.5` 的筛选体系应演进为“状态筛选 + 组筛选”双维度并存，而不是只在现有单一筛选上继续打补丁。
- 第二版手工验收应落到统一清单 `docs/V015_ACCEPTANCE.md` 中执行与勾选，避免再次以口头描述替代验收基线。
- 当前开发版与安装版共享同一应用标识 `com.qingdan.desktop`，因此默认共用同一 `AppData` 数据目录与 SQLite 文件。

### 5. `v0.20` 实施边界

- `v0.20` 进入实施前，优先级、更多条件、批量模式和任务概览增强都应先对齐到 `docs/ARCHITECTURE.md` 与 `docs/V020_ACCEPTANCE.md`。
- 本轮继续坚持文档先行，任何实现前的范围变化都应先回写到文档，再进入代码。
- `v0.20` 仍不改变 GitHub Release 默认必须同时附带 `setup.exe` 与 `zh-CN.msi` 的规则。
- 若实施过程中出现与既有收口规则冲突的诉求，优先保留已收口规则，再单独讨论新增版本边界。
